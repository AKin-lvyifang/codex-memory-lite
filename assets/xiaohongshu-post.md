【标题备选】

1. Codex 会话一长就忘？我给它装了个长期脑子
2. 我没换模型，只给 Codex 加了套外部记忆
3. 长项目总失忆？这套结构化记忆真的能救

【正文】

真的被 Codex 长会话失忆折腾过很多次……

前面刚定好的规范，后面聊久了就忘。
自动压缩上下文吧，又容易把关键事压没。
新线程再继续做的时候，还常常得重新解释背景。

所以这次我给它做了一套“结构化记忆”。

📌 它是干什么的？
就是把 Codex 原来容易丢的项目上下文，拆成几类单独保存。
不是所有东西再堆一页里，而是分开记。

📌 它怎么分？
当前正在做什么，放一份
长期不会变的规则，放一份
每个大任务，再单独建自己的记录
历史过程和旧信息，另外归档

📌 它怎么工作？
根规则负责判断：这个项目是不是已经够复杂，该启用结构化记忆了
项目规则负责规定：新会话进来先读哪里
3 个 skills 负责自动建目录、建任务、做阶段同步

📌 这样有什么效果？
新会话不会一上来就懵
旧规范不容易丢
任务不会串台
历史也能查，但不会继续污染当前状态
因为默认只读必要入口文件，所以 token 压力也会明显更小

📌 说白了它的原理是什么？
就是把“只活在聊天里的记忆”
变成“项目自己带着走的外部记忆”。

不是让模型更聪明，
而是让它少忘事。

📌 “跨项目”怎么理解？
这个也要讲清楚。
它最强的是同一个项目里的跨会话、跨线程续跑。
到了跨项目层面，更适合复用统一规则、统一模板、统一 skills，
而不是不同项目自动共享全部业务上下文。

如果你也在用 Codex 跑长期项目，这套真的很值得搞。
