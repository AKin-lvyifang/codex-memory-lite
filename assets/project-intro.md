# Codex 结构化记忆

## 这是什么

这是一个给 Codex 用的“项目外部记忆层”。

它不是换模型，也不是换提示词大法。

它做的事情很简单：

- 把原本只存在聊天里的重要上下文
- 变成项目文件里可复用、可续接、可分层读取的记忆文件

## 它解决什么问题

以前最常见的问题是：

- 会话一长，上下文越来越大
- 自动压缩后，前面定过的规范和决定容易丢
- 新线程继续做时，常常又要重新解释背景
- 越到后面，越多 token 被花在“回忆”和“复述”上

这套方案解决的不是“模型不聪明”。

而是“长期项目的记忆结构不对”。

## 它是怎么工作的

核心只有一句话：

不要把所有信息都堆在一个 handoff 文件里。

而是拆成 4 层：

- `current.md`
  - 只放当前有效状态
- `spec/`
  - 放长期稳定规则
- `tasks/`
  - 每个大任务单独记
- `archive/`
  - 放历史过程和旧信息

然后再配 3 个 skills：

- 初始化项目记忆
- 初始化任务记忆
- 在阶段切换时同步记忆

## 它说的“跨项目”到底是什么意思

这里要讲清楚，不然很容易吹过头。

这套方案最强的是：

- 同一个项目里的跨会话
- 同一个项目里的跨线程续跑

到了跨项目层面，它能做到的是：

- 让不同项目复用同一套规则
- 复用同一套模板
- 复用同一套 skills

也就是复用“记忆机制”。

不是让不同项目自动共享全部业务上下文。

## 最后的效果

- 新会话一进来，不会先读一大坨旧历史
- 当前在做什么，会很快进入状态
- 老规范不容易丢
- 不同任务不会串台
- 历史还能查，但不会继续污染当前上下文
- 因为默认只读必要入口文件，所以 token 压力也会更小

说白了：

这不是给 Codex 增加智商。

而是给它装一个“不容易忘事的长期脑子”。
